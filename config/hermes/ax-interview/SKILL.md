# 의료진 AX 워크플로우 심층 인터뷰 스킬

> 본 스킬은 **인터뷰 작업의 구체적 절차(SOP)**만을 정의합니다. 에이전트의 정체성·대화 원칙(One Question at a Time, AI 에이전트 용어 엄수, No Rushing, 임상 데이터 심층 추출 태도, 스레드/1:1 작동 원칙)은 `/opt/data/SOUL.md`에 정의되어 있으며 항상 함께 적용됩니다. 본 문서와 SOUL이 충돌하는 것처럼 보이면 **SOUL의 대화 원칙이 우선**입니다.

---

## 1. 트리거 및 시작

- **트리거**: Mattermost 인터뷰 채널에 사용자가 **루트 게시글**로 '아이디어 명칭(업무 아이디어)'을 게시한 순간.
- 루트 게시글의 텍스트가 곧 **Step 1의 대상 업무명**입니다. 중복해서 업무명을 묻지 마십시오.
- 첫 답글 예시 — *"작성해주신 **[아이디어 명칭]**에 대한 AX 워크플로우 심층 인터뷰를 이 스레드에서 시작하겠습니다. 저는 AX 추진단의 AI 에이전트 인터뷰어입니다. 먼저 선생님의 **진료과**와 **주 업무 병동/소속**을 여쭤봐도 될까요?"*

---

## 2. 인터뷰 흐름 (5단계)

### Step 1. 대상 업무 정의
- 루트 게시글의 아이디어 명칭을 대상 업무명으로 채택 (재질문 금지).
- **진료과** 및 **주 업무 병동/소속**(예: OOO과 / OO병동 / HOD 수) 확인.
- 해당 대상 업무의 **일/주간 발생 빈도** 도출.

### Step 2. 현행 업무(AS-IS) 해부 — Input · Process · Output
- **[Trigger]** 업무 시작 계기 및 시작 시점.
- **[Input]** 업무 시작 시 가장 먼저 확인하는 시스템 화면·정보 소스.
- **[Process] (행동/판단 초정밀 분해 — ⚠️집요하게 파고들 것)**
  - 인터뷰이의 머릿속 판단과 손가락의 움직임을 완벽히 분리.
  - *나쁜 예*: "하나씩 클릭해서 봅니다" → "그렇군요. 다음 산출물은 무엇인가요?" (X)
  - *좋은 예*: "하나씩 클릭해서 봅니다" → "아, 창을 여러 개 띄우셔야 하는군요. EMR 메인에서 간호기록, Lab, V/S 화면을 각각 따로 여시려면 한 환자당 대략 몇 번 정도 클릭하나요? 수치를 메모장에 따로 적어두시나요?" (O)
- **[Output]** 최종적으로 시스템에 남는 산출물(메모, 처방 변경 등).
- **소요 시간 & 주요 시스템**: 1건(또는 1회) 처리 시 소요 시간 및 사용 시스템(EMR, OCS 등).

### Step 3. Pain Point 도출

이 단계도 Step 4와 동일하게 **에이전트가 먼저 후보 Pain Point를 짚어내고 인터뷰이가 검증·수정하는 방식**으로 진행합니다. 의료진은 자신의 워크플로우에 이미 익숙해져 있어 "특별히 불편한 건 없다"고 답하기 쉬우므로, Step 2(AS-IS)에서 드러난 행동 패턴을 근거로 에이전트가 먼저 의심 구간을 짚어 주어야 합니다.

- **Pain Point 사전 제시 (⚠️ 핵심)**
  - Step 2에서 확인한 구체적 행동(클릭 횟수, 화면 전환 빈도, 수동 메모, 타 부서 연락 대기, 수치 비교 등)을 근거로 에이전트가 먼저 **"방금 들어보니 환자별로 EMR 창을 ○회 전환하시고, Lab 값을 메모장에 따로 적으시는 부분이 보였는데 — 이 구간이 가장 시간을 잡아먹거나 놓치기 쉬운 지점 아닐까요?"** 형태로 1~3개의 의심 Pain Point를 짚어 제시합니다.
  - 인터뷰이가 명확하게 불편 구간을 먼저 표현했다면 그 내용을 우선 반영하되, **"별로 없다", "익숙해서 괜찮다"** 수준의 답변일 경우 본 사전 제시 방식으로 잠재 Pain Point를 끌어냅니다.
  - 제시는 반드시 **Step 2의 실제 행동에 근거**해야 하며, 일반론(예: "EMR이 불편하시지요?")은 금지합니다. 인터뷰이가 부인·수정·우선순위 재조정을 하면 그 반응 자체가 새로운 정보이므로 꼬리 질문으로 이어갑니다.
- **Pain Point 카테고리 (체크리스트)**: 아래 유형 중 어디에 해당하는지 함께 확인합니다 (보고서 2절의 체크박스와 1:1 대응).
  - 여러 화면/기록을 뒤져보는 시간 (정보 파악 지연)
  - 단순 텍스트 반복 타이핑 (기록 피로도)
  - 타 부서 연락/대기 시간 (커뮤니케이션 병목)
  - 복잡한 조건(수치 등) 수동 확인 (휴먼 에러 위험)
  - 기타
- **확정된 Pain Point**: 인터뷰이의 검증을 거쳐 합의된 핵심 Pain Point와 구체적 예시(언제·어떤 상황에서 발생하는지)를 정리합니다.

### Step 4. TO-BE 설계 및 기대효과

이 단계는 **에이전트가 먼저 설계 초안을 제안하고 인터뷰이가 검증·수정하는 방식**으로 진행합니다. 의료진은 "AI에게 무엇을 시키고 싶은가?"라는 열린 질문에 답하기 어려워하는 경우가 많으므로, AS-IS에서 드러난 병목·반복 행동을 근거로 에이전트가 먼저 구체적인 AX 포인트를 그려 보여야 합니다.

- **AX 설계안 사전 제안 (⚠️ 핵심 변경)**
  - Step 2(AS-IS)와 Step 3(Pain Point)에서 수집한 행동·판단·병목을 토대로, 에이전트가 먼저 **"제가 AS-IS를 들어보니 이러이러한 부분(예: 환자별 EMR 다중 창 전환 클릭 N회, Lab Trend 수동 확인)이 AX 후보로 보입니다. 이런 식으로 AI 에이전트가 대신 처리하는 안을 생각해봤는데 어떠신가요?"** 형태로 1~3개의 구체적 설계안을 먼저 제시합니다.
  - 인터뷰이가 명확한 AX 의도를 이미 표현했다면 그 의도를 우선 반영하되, **모호하거나 "AI가 알아서 해주면 좋겠다" 수준의 답변**일 경우 본 사전 제안 방식을 활용해 AX 포인트를 끌어냅니다.
  - 제안은 반드시 **AS-IS의 실제 행동에 근거**해야 하며, 일반론(예: "AI가 진료를 도울 수 있습니다")은 금지합니다. 인터뷰이가 거부·수정하면 그 반응 자체가 새로운 정보이므로 꼬리 질문으로 이어갑니다.
- **AI 에이전트의 구체적 임무 (확정안)**: 위 제안과 인터뷰이 피드백을 통합해 AI가 담당할 영역을 정의 (정해진 시간 자동 취합, 우선순위 정렬, 대시보드 생성 등).
  - **추가 확장 기능 적극 발굴**: 환자 상태 변화 패턴 기반 위험 예측, 이전 데이터 대비 이상 패턴 탐지, 중요 이벤트 자동 요약, 위험 환자 우선순위 추천 등.
- **사람(의사)의 개입 지점**: 에이전트의 취합/요약/추천 결과를 의사가 최종 판단 및 OCS 처방 입력으로 연결하는 인간-AI 협업 시나리오 구체화.
- **최종 산출물 활용 방법 (💡 신규)**: AI 에이전트가 생성한 **최종 산출물을 어떤 형식으로 받아 어떻게 활용하고 싶은지** 구체적으로 도출합니다. 산출물의 "형태"와 "소비 시점·방법"을 모두 묻습니다.
  - **형식 후보 예시**: 대시보드 화면(EMR 내 임베드 or 별도 웹), 회진 직전 자동 생성되는 환자별 요약 카드(PDF/마크다운), Mattermost·카카오톡 등으로 푸시되는 알림 메시지, OCS에 자동 입력 가능한 처방 초안 텍스트, 모바일에서 조회 가능한 리스트 뷰 등.
  - **활용 시나리오 질문 예시**: "회진 직전 5분에 휴대폰으로 확인하시고 싶으신가요, 아니면 EMR 로그인 직후 자동으로 첫 화면에 떠 있길 원하시나요?", "산출물을 그대로 처방·기록에 복사해 쓰실 건가요, 아니면 참고용으로만 보고 직접 다시 입력하시는 게 편하신가요?"
  - 산출물 형식이 정해져야 데이터 요구사항(Step 5)의 출력 단위(환자당 1건 vs 병동 단위 집계 등)가 결정되므로, 본 항목을 모호한 채로 넘기지 마십시오.
- **도입 시급성·영향도(우선순위)**: 즉시 도입 필요 / 도입 시 편리함 / 천천히 도입 중 선택.
- **도입 시 예상 효과**: 정량적(시간 단축 % 등) + 정성적(환자 안전성, 진료 퀄리티 향상 등).

### Step 5. 상세 데이터 요구사항 (Data Requirements — 초정밀화)

TO-BE 산출물(Step 4에서 정의한 형식)을 생성하기 위해 실시간 수집해야 하는 데이터 사양을 **`구분 → 조사 항목`** 트리 구조로 확인합니다. 각 항목은 "어떤 데이터를 / 어떤 비교·필터링 기준으로 / 어떤 예시 값을" 추출할지까지 구체화해야 합니다 (`보고서 4절`의 표 구조와 1:1 대응).

| 구분 | 조사 항목 | 확인 포인트 (예시 답변 유도) |
| :--- | :--- | :--- |
| **진단** | 주진단 | 환자별 주진단명 (예: Pneumothorax) |
| **진단** | 부진단 | 동반 진단명 (예: COPD) |
| **검사** | 진단검사 | slip별 Trend 조회에서 이상치 확인, 최근값으로 비교 *(예: `[5.18] Routine CBC : 12.30 x10³/㎕, CRP 9.33mg/dL`)* |
| **검사** | 진단검사_미생물 | 미생물검사 환자별 결과 조회 *(예: `Fungus Culture : Candida albicans`)* |
| **검사** | 영상검사 | 영상검사 판독 결과, 최근 판독으로 비교 *(예: `[5.19] Chest PA — Slightly decreasing small left pneumothorax …`)*. **체크포인트**: 전 영상과 비교 가능하도록 판독되고 있는가? |
| **검사** | 병리검사 | 상태가 '실시'인 경우 **결과보고 예정**임을 함께 표기 |
| **검사** | 기타검사 | 기타검사 결과 조회, 최근 판독으로 비교 *(예: `[5.15] EKG 검사결과`)* |
| **수술** | 수술내역 | 수술 예정 내역 포함하여 조회 |
| **처방** | 약·검사·처치 등 | 처방된 약·검사·처치의 **실제 실시 여부** 확인 *(예: `O2 6L/min`)* |
| **TPR** | V/S | 정상치에서 벗어난 이상 V/S 확인, **V/S 트렌드 해석 기준** |
| **진료기록** | EMR | 이전 경과기록 대비 **변동사항** |
| **진료기록** | 간호기록 | 의사에게 보고된 기록, 호흡곤란·피부통합성 장애 등 **환자 증상 기록**이 있는 경우 확인 |
| **진료기록** | 협진기록 | 협진 **회신 여부** 및 **회신 내용** 확인 |

- 위 표는 **인터뷰 진행 시의 질문 체크리스트**이자 **보고서 4절의 출력 스키마**입니다. 빈 칸 없이 모든 행을 채우는 것을 목표로 하되, 인터뷰이의 업무 성격상 해당 없는 항목은 *"해당 없음"* 으로 명시합니다.
- 항목별로 한 줄 답변에 그치지 말고 **"어떤 수치인가 → 그 수치를 어떤 기준으로 판단하는가 → 이상치/이벤트가 보이면 그다음 어떤 행동으로 이어지는가"** 를 꼬리 질문으로 끌어내십시오 (SOUL의 임상 데이터 심층 추출 원칙 적용).

## 3. 최종 보고서 작성 및 업로드

인터뷰이가 '인터뷰 종료'를 요청하거나 Step 5까지의 정보가 충분히 수집되었다고 판단되면 대화를 정중히 마무리하고 다음 절차를 자동 수행합니다.

### 3.1 보고서 본문 작성
**먼저 보고서 양식 파일을 읽어와** 그 구조를 그대로 유지한 채 수집된 인터뷰 데이터로 채워 넣습니다. 구조·표 서식·주의사항 멘트·가이드를 임의로 제거하거나 변형하지 마십시오.

- 양식 파일: `/opt/data/skills/ax-interview/report_template.md`

### 3.2 파일 저장
완성된 마크다운을 다음 경로로 저장합니다.

- **경로**: `/opt/data/workspace/[작성일자]-[사용자명]-[아이디어명].md`
- **예시**: `/opt/data/workspace/2026-05-27-신성철-환자_요약_워크플로우.md`
- **명명 규칙**
  - `작성일자` — 오늘 날짜(`YYYY-MM-DD`).
  - `사용자명` — 인터뷰이의 한글 이름(예: `신성철`).
  - `아이디어명` — 루트 게시글의 업무 아이디어 명칭(예: `환자_요약_워크플로우`).
  - 특수문자·띄어쓰기는 하이픈(`-`) 또는 언더바(`_`)로 치환.

```python
content = """[수집된 정보를 양식에 채워 넣은 마크다운 본문]"""
with open("/opt/data/workspace/2026-05-27-신성철-환자_요약_워크플로우.md", "w", encoding="utf-8") as f:
    f.write(content)
```

### 3.3 Mattermost 업로드
저장 즉시 헬퍼 스크립트로 **현재 인터뷰 스레드**에 파일을 업로드합니다.

```bash
python3 /opt/data/skills/ax-interview/scripts/upload_report.py \
  --file "/opt/data/workspace/[작성일자]-[사용자명]-[아이디어명].md" \
  --channel "${MATTERMOST_CHANNEL_ID}" \
  --root "${MATTERMOST_ROOT_ID}"
```

- 대화 컨텍스트에서 채널 ID 및 루트 스레드 ID를 파악해 인자로 매핑합니다.
- 메타데이터를 얻지 못했다면 `--channel`·`--root` 인자를 생략하여 환경변수에 지정된 홈 채널로 전송되도록 합니다.

### 3.4 마무리 인사
업로드 완료 후 인터뷰이에게 "분석이 완료되어 정식 보고서를 파일로 첨부해드렸습니다." 와 같이 정중히 완료 보고를 드리고 인터뷰를 마칩니다.
